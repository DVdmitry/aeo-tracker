## Folded-in additions (must implement)

These were proposed in review and accepted by the user. They are part of v0.5 scope, not optional.

### P1 — Explicit category question in `init`

Before LLM auto-suggest, prompt the user:
```
What does your company do? (one sentence — used to disambiguate ambiguous categories)
> _
```
Pass this string into the brainstorm prompt as `CATEGORY_DESCRIPTION`. **Single most important addition** — it solves the webappski-class problem (AEO ambiguity in Poland) at the source.

In non-interactive mode (`--yes`): require `--category="..."` flag. If missing, error with: *"--yes mode requires --category to disambiguate (e.g., --category='AI search visibility for SaaS')"*.

### P2 — Bring-your-own-keywords mode

```bash
aeo-tracker init --keywords="best X 2026,how to Y,X for Z" --brand=X --domain=Y --yes
```
Skips brainstorm entirely. Zero LLM calls. For users with their own keyword research workflow.

### P3 — Cross-model validation

Phase 4 (validation) must use a **different provider** than Phase 1 (brainstorm). If brainstorm = Anthropic, validation = OpenAI, and vice-versa. If only one LLM provider configured: skip cross-model validation with warning *"Cross-model validation skipped (only one LLM provider available). Single-model bias risk."*

### P4 — Disambiguation warning in `cmdReport`

In `lib/report/sections.js`, add a new section that triggers when:
- Score is 0% across all queries, AND
- Top canonical sources don't contain brand-related keywords (heuristic: none of top 5 sources include any brand/domain token or the user's category description)

Render warning:
```
⚠ Your queries may be matching the wrong industry.
Top cited URLs ({list}) suggest AI engines interpret your queries differently than expected.
Consider: aeo-tracker init --refresh-keywords --category="<more specific>"
```

### P5 — Optional opt-in telemetry

In `init` after final accept:
```
[Optional] Share anonymized query suggestions back to webappski to improve future versions?
Sends: site language, generated queries, your acceptance rate. NEVER: brand, domain, site content.
[y/N, default: N]: _
```
If accepted: write to `~/.aeo-tracker-telemetry/<date>.json` for now. Defer actual upload mechanism to v0.5.1.

**If implementation requires more than ~50 lines, defer P5 entirely to v0.5.1** and note the deferral in this document.

## Additional architectural requirements

### Ambiguity check moves into LLM prompt

Add to brainstorm prompt:
```
For each generated query, ask yourself: "Could any acronym or term in this query mean something else in {site_language/region}?"
If yes — expand the acronym OR add a disambiguating context word.
Output format: include "ambiguity_note" field per query when applicable.
```

The hardcoded `AMBIGUOUS_ACRONYMS` list stays as a **belt-and-suspenders client-side check** but is no longer the primary mechanism.

### Phase logging for debugging

Each phase prints: `Phase N (name): success / fail / skipped — $cost — duration ms`. Without this, multi-phase failures are undebuggable.

### `--light` flag for cheap mode

`aeo-tracker init --light` runs the **single-shot path** (current 0.4.x behavior, ~$0.0024). The full pipeline runs by default (~$0.04). For power users who want speed/cost over thoroughness.

### Decision-fatigue UX

Default display: **top 3 + button to "show 5 alternatives"**. Not 12 candidates immediately. Add `[m] manual — type my own queries` to the accept menu.

### Cost structure to document in README

| Mode | Cost | Use when |
|---|---|---|
| `--light` | ~$0.0024 | Power users, CI re-runs |
| Default (full pipeline) | ~$0.04 | First-time setup, max quality |
| `--keywords=...` | $0 | BYO keywords |

Before declaring v0.5 ready: **benchmark on 5 reference brands**:
- TypelessForm

- Notion

**Target: ≥7/15 matches (47%)** across the 5 brands. Below this — pipeline isn't an improvement worth shipping; iterate prompt engineering before merge.

Cost of benchmark: ~$0.20 ($0.04 × 5 brands). Use real-data, not mocks.

### C7 — Selection + decision-fatigue UX (top 3 + show 5 alternatives + manual fallback)

**Guard 3 — Intent-diverse selection (blocking).** Algorithm must pick exactly 1 candidate per intent bucket: 1 commercial + 1 informational + 1 vertical. **NOT** top-3 by overall score. If any required bucket is empty after filter+validation, fall back to adjacent bucket with explicit warning. Reason: a tracker measuring only commercial-intent queries misses a whole dimension of user journeys.

**Guard 4 — Alternatives display hygiene.** Selected top-3 must all be validated (onCategory=true, high confidence). The 5 alternatives shown for swap can be unvalidated (scored only), but MUST carry a visible `(unverified)` label. Never silently present unvalidated candidates as equivalent to validated ones.

**Guard 5 — Audience-qualifier stripping in validation prompt (blocking).** When CATEGORY_DESCRIPTION narrows to an audience (e.g., "AEO services for SaaS"), the validation prompt must strip the audience qualifier before asking validator to judge vertical-bucket candidates. Otherwise validator rejects legitimate vertical expansion ("AEO for healthcare" when category said "for SaaS"). Test case: `category="AEO services for SaaS"` + candidate `"Answer Engine Optimization for healthcare providers"` → must PASS validation, not reject.

**Guard 6 — Per-intent scoring if vertical dominance detected.** If during C7 verification the vertical bucket consistently wins top-score positions (≥4 of top 5), split scoring features: commercial-specific bonuses (recency, pricing markers), informational-specific bonuses (how-to / what-is patterns), vertical-specific bonuses (specificity markers). Prevents vertical-specificity winning all selection slots.

## Open issues encountered during implementation

Log unexpected issues here. Helps post-mortem and informs v0.6.0 planning.

- **Competitor list disambiguation (defer to C7).** C2 verification on webappski generated competitors like "AEO Monitor" and "AEO Check" — short branded names containing "AEO". Same ambiguity concern applies to competitor lists as to queries: in Poland "AEO Check" could match a customs-checking service. Apply the same category-aware expansion logic to competitors during C7 (selection + review UX).
- **`detectAudience` and `detectGeography` are dead code until C3.** C1 added both but nothing consumes them yet. They will be fed into the Phase 2 brainstorm prompt in C3 to produce vertical-intent queries with specific audience tags. Not a bug — just flagged so nobody thinks they don't work.
- **Minimum length validation for `--category` (fix in C8 or C13).** Currently `--category="x"` is accepted with no warning. Best practice: require ≥20 characters, warn if shorter ("category description seems very short — may not disambiguate effectively"). Low priority — user who passes 1-char category is doing it wrong anyway.
- **Vertical-intent over-narrowing observed in Mode 3 (TypelessForm).** Q3 generated as "voice form autofill widget for hotel and travel booking forms" — locked to one vertical (hospitality) when TypelessForm serves all verticals. C3 brainstorm must produce vertical-intent candidates across ≥3 different industries (healthcare, finance, e-commerce, etc.); C5 ranking picks one from the diverse pool. Don't let the LLM pick a single vertical and run with it.
- **Validator-vs-Guard-2 tension (surfaced in C6 stress test).** Validator rejected `"Answer Engine Optimization for healthcare providers"` with reason "Healthcare providers are not within the SaaS target audience" — because CATEGORY_DESCRIPTION narrowed to "SaaS". But Guard 2 wants vertical-bucket candidates spanning ≥3 industries. Fix in C7: when CATEGORY_DESCRIPTION narrows to one audience, either (a) broaden the `categoryDescription` used in validation prompt to "tracking Answer Engine Optimization visibility — applicable across SaaS, healthcare, fintech, e-commerce, etc.", or (b) skip cross-model validation for vertical-bucket candidates. Current E2E on webappski did not hit this because user-provided category was "Answer Engine Optimization services for SaaS — NOT customs" and Anthropic brainstorm already matched strictly to SaaS verticals.
- **Competitor hallucination observed in Mode 3 (TypelessForm).** LLM suggested "Tally with voice input", "Typeform voice", "Google Duplex for forms" — none of which are real products with those names. Partially mitigable by cross-model validation in C6; fully mitigable only by post-suggest web-search verification (out of scope for v0.5, note for v0.6).
