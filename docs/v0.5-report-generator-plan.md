# v0.5 — Report Generator (architecture plan)

> Status: design document, not yet implemented.
> Goal: add `aeo-tracker report` command that produces a Week-3-style data report — dry, quantitative, diff-aware — **without narrative, without external dependencies, without code reuse from typelessform**.

---

## 1. Guiding constraints

These are non-negotiable. They define what makes this v0.5 different from every competitor.

| Constraint | Why it matters |
|---|---|
| **Zero runtime dependencies** | Current USP (see README line 20). Adding puppeteer/canvas/chart.js would add 50-100MB and kill the "Node 18+ only" positioning. |
| **Inline SVG only for charts** | SVG renders natively in GitHub, Notion, browsers, PDF converters. No image binaries, no canvas, no headless browsers. |
| **Data report, not narrative** | No "Three Surprises", no "Honest Frame", no editorial tone. Report contains only: numbers, deltas, tables, raw quotes. Narrative is human work. |
| **Output is portable text** | Markdown primary; single-file HTML optional. Both readable without tooling. |
| **No code copied from typelessform** | Separate codebase, separate purpose. Week-3 article charts were one-off; this is a reusable report engine. |
| **Neutral colour palette** | Tool is open-source and used by third-party brands. Hard-code tailwind defaults (`#10b981` / `#ef4444` / `#94a3b8`), never typelessform or webappski brand colours. |
| **Opt-in cost escalation** | Extra API calls (dual-model, Perplexity, etc.) hide behind opt-in config flags. Default `aeo-tracker run` stays at ~$0.05 per run. Users who want drift detection or broader coverage pay only for what they enable. |

---

## 2. New CLI commands

Three additions. Each composable with existing `run` command.

```bash
# v0.2 (already on roadmap)
aeo-tracker diff 2026-04-10 2026-04-17
# Prints diff table between two runs.

# v0.5 — NEW
aeo-tracker report
# Generates weekly-report.md from the latest 2+ runs in aeo-responses/.

aeo-tracker report --html
# Same report, wrapped in a single-file HTML with inline SVG + minimal CSS.

aeo-tracker report --since 2026-03-23
# Multi-week report covering everything from that date forward.
```

**Default output paths:**
- `aeo-reports/2026-04-17/report.md`
- `aeo-reports/2026-04-17/report.html` (if `--html`)
- `aeo-reports/2026-04-17/charts/` (generated SVGs, embedded inline but also saved separately for reuse)

## 3. File structure

Current: everything lives in `bin/aeo-tracker.js` (25KB, one file).

Proposed v0.5 layout — still zero-deps, but modular:

```
aeo-tracker/
├── bin/
│   └── aeo-tracker.js           # CLI entry, arg parsing, command routing (unchanged interface)
├── lib/
│   ├── providers/
│   │   ├── openai.js            # existing, extracted
│   │   ├── gemini.js
│   │   └── anthropic.js
│   ├── parse.js                 # existing mention/source detection
│   ├── diff.js                  # NEW — compare two _summary.json files
│   ├── report/
│   │   ├── markdown.js          # NEW — render report as markdown
│   │   ├── html.js              # NEW — wrap markdown in single-file HTML
│   │   ├── extract-quotes.js    # NEW — pull verbatim mentions from raw responses
│   │   └── sections.js          # NEW — section builders (keep order + structure consistent)
│   └── svg/
│       ├── heatmap.js           # NEW — AI × Query matrix renderer
│       ├── barchart.js          # NEW — competitor mentions, trend bars
│       ├── sparkline.js         # NEW — mini-trend inline in tables
│       └── delta-arrow.js       # NEW — ↑↓ with color-coded delta label
└── package.json
```

All new files pure ESM, no imports outside Node stdlib. Total added code target: ~1500 lines.

## 4. Report structure (section order)

Lifted from Week 3 article **with narrative stripped**. Each section deterministic — no human interpretation.

### 4.1. Header block

```
@webappski/aeo-tracker — Report
Brand: acme
Period: 2026-03-23 → 2026-04-17 (25 days, 4 runs)
Generated: 2026-04-17T14:22:00Z
```

### 4.2. Top numbers block

```markdown
## Report in 5 Numbers

1. **3/3** — mentions on Perplexity (stable, 4 runs)
2. **2/3** — mentions on Gemini 2.5 Pro (up from 0/3 at day 1)
3. **0/3** — mentions on ChatGPT GPT-5.4 (unchanged)
4. **+1381%** — delta since baseline (when mentions exist — else omitted)
5. **0** — net-new competitors entering the answer space this period
```

Rules:
- Always exactly 5 items.
- Skip rows with no meaningful data rather than filling with zeros.
- No editorial phrasing. No "surprises", no "wins", no "challenges".

### 4.3. Per-engine dual-model comparison

One table per provider. Current vs previous model generation (v0.3 feature — integrated here).

```markdown
### ChatGPT

| Model                | Q1  | Q2  | Q3  | Score |
|----------------------|-----|-----|-----|-------|
| GPT-5.4 (current)    | ❌  | ❌  | ❌  | 0/3   |
| GPT-4o-search (prev) | ❌  | ✅#1| ❌  | 1/3   |
```

### 4.4. AI × Query matrix (heatmap SVG)

Inline SVG heatmap. Rows = engines × model, columns = queries, cells = ✅/❌ with color.

**SVG rendering rules:**
- 100% width, viewBox preserves aspect.
- Colors: fixed neutral palette — green (`#10b981`) for mention, red (`#ef4444`) for miss, gray (`#94a3b8`) for not-tested. Never brand colours.
- Cell labels: "Q1", "Q2", "Q3" (config'd from `.aeo-tracker.json`).
- No title, no legend outside SVG — renders cleanly in dark and light modes.

### 4.5. Week-over-week diff (from `aeo-tracker diff`)

```markdown
### What Changed (2026-04-10 → 2026-04-17)

| Change | Provider | Query | Was | Now |
|--------|----------|-------|-----|-----|
| ↑ Gained | Gemini 2.5 Pro | Q1 | ❌ | ✅ #1 |
| ↓ Lost | ChatGPT GPT-5.4 | Q2 | ✅ #1 | ❌ |
| = Stable | Perplexity | Q3 | ✅ | ✅ |
```

Renders with SVG delta arrows (↑ green, ↓ red, = gray).

### 4.6. Competitor movement

```markdown
### Competitor Mentions — This Period vs Previous

| Competitor | Previous | Current | Δ |
|------------|----------|---------|---|
| Asana      | 5        | 7       | +2 |
| Monday.com | 4        | 4       | = |
| Notion     | 3        | 2       | -1 |
| NewEntrant | 0        | 3       | +3 NEW |
```

Bar-chart SVG alongside.

### 4.7. Raw response excerpts (Perplexity-style)

This is the **killer feature** that replicates what made Week 3 citable. Pull verbatim quotes from raw responses:

```markdown
### What AI Engines Actually Said

**Perplexity, Q2 (2026-04-17):**
> "TypelessForm – One-shot voice form-filling widget; drop a <script> tag, users speak once, and AI populates all fields at once. Works with any standard HTML form, supports 25+ languages, and is GDPR-compliant."

**Gemini 2.5 Pro, Q1:**
> "over 25 languages… AI populates fields, user reviews and edits"

**Claude Opus 4.6, Q2:**
> Source cited: /blog/best-ai-form-filling-tools-2026
```

Extraction rules (in `lib/report/extract-quotes.js`):
- Find brand name in response text.
- Extract ±N characters around the match (default 200).
- Break on sentence boundaries (not mid-word).
- Strip markdown formatting noise.
- Include citation URLs if present in the response metadata.

**Citation-only mentions** (brand in cited URL but not in answer text) render as a distinct block — these are a weaker but meaningful signal and must not be conflated with verbatim quotes:

```markdown
**Gemini 2.0 Flash, Q2 — citation only:**
> Brand appears only as a source URL in the answer:
> https://acme.com/blog/why-x-matters
> (the answer text did not reference the brand directly)
```

### 4.8. Trend sparklines (per query)

Inline SVG sparkline next to each query showing mention count over time:

```
Q1: Fill all fields at once with voice   [▁▃▅▆▆]  mentions trending up
Q2: Best tools voice input HTML forms    [▃▃▃▂▁]  trending down
Q3: Voice input fills entire form...     [▁▁▂▃▅]  growing
```

Pure SVG, 80×20px, renders inline in most markdown viewers.

### 4.9. Footer

```
Raw responses: aeo-responses/2026-04-17/
Run this report: aeo-tracker report --since 2026-03-23
aeo-tracker v0.5.0 — @webappski
```

**Intentionally excluded** (vs Week 3 article):
- No "What I Got Wrong" — humans interpret corrections.
- No "Three Surprises" — humans notice surprises.
- No "Honest Frame" or closing narrative — humans write that.
- No soft sell / lead-gen text. The README already does that.

## 5. SVG rendering primitives

All four renderers pure functions — `(data, options) → string`. No DOM, no Canvas.

### 5.1. `heatmap(rows, cols, cells)`

- Input: 2D array of `'hit' | 'miss' | 'na'`.
- Output: `<svg>...</svg>` string.
- Size: responsive `viewBox="0 0 800 200"`, `width="100%"`.
- Each cell = rect + optional text label.

### 5.2. `barchart(labels, values, options)`

- Horizontal bars.
- Auto-scale to max value.
- Deltas rendered as smaller bar of different color.
- ~200 lines.

### 5.3. `sparkline(values)`

- Single polyline through normalized points.
- 80×20 viewport.
- Current value + trend color (green up / red down / gray flat).
- ~50 lines.

### 5.4. `deltaArrow(value, direction)`

- Tiny inline SVG: ↑ / ↓ / = with color.
- Used in diff tables.
- ~20 lines.

All SVGs embed inline in markdown via `<svg>...</svg>` blocks. GitHub and most markdown renderers accept inline SVG.

## 6. Data flow

```
aeo-responses/
├── 2026-03-31/
│   ├── q1-openai.json
│   ├── q1-gemini.json
│   ├── ...
│   └── _summary.json
├── 2026-04-07/
├── 2026-04-10/
└── 2026-04-17/

                            ↓  aeo-tracker report

                  ┌──────────────────────┐
                  │ lib/diff.js          │
                  │ read 2+ _summary.json│
                  │ compute deltas       │
                  └──────────────────────┘
                            ↓
                  ┌──────────────────────┐
                  │ lib/report/          │
                  │ extract-quotes.js    │
                  │ read raw q*.json     │
                  │ pull verbatim quotes │
                  └──────────────────────┘
                            ↓
                  ┌──────────────────────┐
                  │ lib/svg/*.js         │
                  │ generate charts      │
                  └──────────────────────┘
                            ↓
                  ┌──────────────────────┐
                  │ lib/report/          │
                  │ markdown.js OR       │
                  │ html.js              │
                  └──────────────────────┘
                            ↓
                  aeo-reports/2026-04-17/report.{md,html}
```

No network, no external processes. Pure Node stdlib: `fs`, `path`, `url`.

## 7. HTML output (optional, `--html` flag)

Single self-contained file. No external CSS/JS.

- `<style>` block: ~100 lines of minimal CSS, prefers-color-scheme aware.
- Inline SVGs untouched.
- Same markdown converted to HTML via a stdlib-only converter (~300 lines, supports just the subset we emit: headers, tables, lists, blockquotes, inline SVG).
- Opens in any browser, renders identically everywhere.

**Use case:** WebAppSki emails the HTML to a client. Client opens it. Looks like a polished report. No download of a 10MB PDF.

## 8. Implementation phases

Split into 3 PRs, each independently shippable:

### Phase A — v0.3.0 business-value features (foundation)

Scope revised after process-level pushback: Phase A no longer leads with dual-model infrastructure. It leads with features that change what the tool _sees_ (Perplexity, explicit competitors, canonical sources) and how it _integrates_ (diff command, `--json` flag, rich exit codes). Dual-model moves to Phase B because it doubles cost without changing what gets measured, only how precisely.

- Extract modules into `lib/` (done in Commit 1).
- Deprecation warning on current `report` command (stays live, replaced in Phase B).
- Explicit `competitors` list in config → direct name detection → competitive intelligence in summaries.
- Canonical sources tracking across runs → "pages AI cites for your vertical" aggregation.
- Perplexity via Sonar API (~$0.005 per query) — closes the biggest coverage gap.
- `aeo-tracker diff` command — compare two runs, show delta table.
- `--json` flag on `run` for CI pipelines.
- Rich exit codes (0/1/2/3) driven by `regressionThreshold` config.
- Latency per provider captured in `_summary.json`.

Estimated: ~550 lines net across 6 commits. Ships as **v0.3.0**. No major bump — schema changes are additive and non-breaking.

See [phase-a-implementation.md](./phase-a-implementation.md) for the per-commit breakdown.

### Phase B — v0.4.0 dual-model + new report engine

- Opt-in `previousModel` per provider — additive schema extension, non-breaking. Default users stay single-model; drift-detection is explicit.
- `lib/svg/*` four renderers (heatmap, barchart, sparkline, delta-arrow).
- `lib/report/extract-quotes.js` for verbatim mentions.
- `lib/report/markdown.js` for the new report surface.
- `lib/report/sections.js` — deterministic section builders.
- Replace `bin/aeo-tracker.js`'s `cmdReport` with the new markdown-first engine. Old HTML/Chart.js path is deleted; deprecation warning from 0.3.0 has had a release cycle to notify users.

Estimated: 800-1000 lines. Ships as **v0.4.0**.

### Phase C — v0.5.0 HTML wrapper

- `lib/report/html.js` with inline CSS.
- Minimal markdown-to-HTML converter (stdlib only).
- `--html` flag handling.

Estimated: 400-500 lines. Ships as **v0.5.0** (additive feature, non-breaking).

### v1.0.0 — later

Not Phase D. v1.0.0 is reserved for the moment the API has been exercised in the field for 2-3 months post-0.5.0 without a breaking change. Staying on 0.x while the public surface is still moving is honest — semver `0.x` signals "we reserve the right to break things", and paying that freedom back with premature 1.0.0 buys us nothing.

## 9. Strategic role

**AEO Tracker has two audiences today:**
1. Developers running weekly checks (`aeo-tracker run` output = enough).
2. **WebAppSki prospects** discovering the tool → reading the README → eventually booking a consultation.

v0.5 reports shift the centre of gravity toward audience #2:

- A client runs `aeo-tracker report --html` once → sees a polished, citation-rich report.
- The report makes the gap visible: *"here's where you are, here's where competitors are"*.
- The implicit question: *"how do I close the gap?"*.
- The README's "About the Methodology" line applies: **results public, playbook sold**.

This turns the tool from a dev-utility into a **lead-generation asset** without adding a single marketing feature.

## 10. What this plan explicitly rejects

For the record — requests that might come later but should be resisted:

| Rejected | Reason |
|---|---|
| PNG/PDF export via puppeteer | Breaks zero-deps USP |
| Canvas/chart.js for richer charts | Breaks zero-deps USP |
| Narrative generation (LLM writes "surprises") | That's what the human does — and what WebAppSki charges for |
| Hosted dashboard | "No third-party dashboards" is a core marketing pillar (README line 3) |
| Importing chart scripts from typelessform | Two codebases, two purposes. Forking creates maintenance debt |
| Emoji stats / marketing fluff in reports | Reports are dry. Marketing lives in README and blog. |

## 11. Success criteria

The v0.5 report is working when:

1. A user who has **never read Week 3 article** gets the same amount of insight from `aeo-tracker report` output.
2. The markdown file renders correctly on GitHub, Notion, and as `cat report.md` in a terminal (SVG falls back gracefully).
3. Total runtime dependencies still **zero**.
4. Full report generation for 4 runs completes in **under 2 seconds** on a laptop.
5. A WebAppSki prospect can paste the HTML report into a Slack DM and the recipient sees charts without downloading anything.

## 12. Resolved decisions

The five open questions have been resolved as follows. Implementation must adhere to these decisions unless explicitly re-opened.

**D1 — Colour palette.** Fixed neutral tailwind palette (`#10b981` / `#ef4444` / `#94a3b8`). No brand colours ever. The tool is open-source and used by third parties — brand-coloured charts would be confusing and unprofessional in their context.

**D2 — Schema change (REVISED).** No major bump now. All 0.3.0 schema changes are **additive and non-breaking**: new optional fields (`competitors`, `canonicalCitations`, `elapsedMs`, `regressionThreshold`) are read if present, absent if not. Dual-model in 0.4.0 adds `previousModel` as an **opt-in** config field — a user who doesn't set it runs single-model as before. v1.0.0 is deferred until 2-3 months of stable public surface. Rationale: 0.x semver explicitly licenses breaking changes, and we are still discovering what the tool needs to be.

**D3 — Multi-brand support.** Deferred to v1.2 (post-v1.1 HTML). Current workaround: one repo / one `.aeo-tracker.json` per brand. An agency running 10 clients keeps 10 folders. This is acceptable ergonomics at v1.0 — we are not optimizing for an agency workflow yet.

**D4 — First-run behaviour.** Graceful degradation. When fewer than 2 runs exist, the diff section is omitted with a short inline note (`_Diff disabled — only one run available. Run weekly to enable week-over-week comparison._`). The report still generates with single-run data: top numbers, heatmap, raw quotes, sparklines (reduced to dots).

**D5 — Citation-only mentions.** Render as a distinct sub-block inside section 4.7 (see updated spec in that section). Never conflate with verbatim quotes — it is a weaker signal, and honesty about signal strength is a core product principle.

## 13. Follow-up questions (not yet asked)

Items that will need decisions during implementation but are not blockers today:

- **Rate limits / retries per provider.** Current code handles this; dual-model doubles the request volume. Exponential backoff or fail-fast?
- **`aeo-responses/` directory growth.** After 52 weekly runs with 3 providers × 2 models × 3 queries = ~936 JSON files/year. Compression? Rotation policy? Or leave as-is (disk is cheap)?
- **Character encoding of raw responses.** Some AI responses contain emoji, RTL, zero-width chars. SVG text rendering must survive all of it.

These surface during Phase B/C and will be decided in code review.
