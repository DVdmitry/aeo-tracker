# v0.5 — Keyword Research Pipeline (full plan)

> Status: design document, not yet implemented.
> Predecessor: v0.4.0 ships with a single-shot LLM auto-suggest that "guesses" 3 queries. This document describes the replacement — a multi-phase pipeline that brainstorms, filters, scores, validates, and selects queries the way an SEO researcher would.
> Target release: **v0.5.0** (separate from 0.4.0 — do not publish 0.4.0 until v0.5 scope is agreed, per user directive).

---

## 1. Why this exists — the webappski case

In 0.4.0, auto-suggest made ONE LLM call with a short prompt and accepted whatever came back. Applied to `webappski.com`, it produced:

- Q1: *"best AEO agency 2026"*
- Q2: *"how to measure AI search visibility"*
- Q3: *"AEO consultants Poland"*

All three contain the acronym "AEO". The site is about **Answer Engine Optimization**. But in Poland and the EU, **AEO commonly means "Authorized Economic Operator"** — a customs-compliance status unrelated to AI.

Running these queries revealed the problem: AI engines returned results from the customs industry (aeodirectory.com, pozwolenieaeo.pl, mddp.pl), not from the user's actual category. The tool was silently measuring the **wrong industry**.

**Root cause:** the tool treated query generation as a trivial step. It isn't — it's the most important configuration decision a user makes, and doing it well requires research and ranking, not guesswork.

v0.5 fixes this by reframing query generation as **keyword research**: brainstorm many candidates, filter out ambiguity, rank by intent coverage and specificity, validate category match, then present a curated selection with alternatives.

---

## 2. Goals and non-goals

### Goals

- Generate **3 tracking queries per intent bucket** (commercial, informational, vertical) that reliably return results from the user's actual category, not a semantic neighbour.
- Surface a **candidate pool** of 10–20 alternatives so the user can swap without a new LLM call if the initial selection doesn't resonate.
- Detect and automatically expand ambiguous acronyms during generation (not only after).
- Validate category match for top candidates via a second LLM pass before showing them to the user.
- Persist the candidate pool in `.aeo-tracker.json` so `aeo-tracker init --swap Q1` can regenerate selection locally without cost.
- Language and geography awareness — queries match the site's language and acknowledge geographic term conflicts (like the AEO case).

### Non-goals

- **Not** integrating SEO volume data from Ahrefs/SEMrush APIs. AI visibility isn't tracked by those tools; there's no meaningful proxy for "AI search volume" yet.
- **Not** building an interactive "swap UI" that requires terminal escape-code manipulation. Edit/swap is a prompt-based flow — pick-a-number style.
- **Not** generating query variations programmatically (permutations of words). We rely on the LLM to produce meaningfully-distinct candidates.
- **Not** running each candidate through all providers to measure before selection — that's the cost of `aeo-tracker run`, not `init`. Validation uses a single LLM call, not N calls.
- **Not** deep NLP analysis (embeddings, semantic clustering). Keep it simple: regex + keyword matching + LLM-based category-check.

## 3. Pipeline architecture — 6 phases

```
┌────────────────────────────────────────────────────────────────┐
│ Phase 1  Site analysis              [existing, reuse]          │
│          Fetch URL, extract meta/headings/body text            │
└────────────────────────────────────────────────────────────────┘
                             │
                             ▼
┌────────────────────────────────────────────────────────────────┐
│ Phase 2  Brainstorm                 [LLM call #1, ~$0.025]     │
│          Generate 25 candidate queries across 5 intent buckets │
│          (commercial, informational, vertical, problem, compar)│
└────────────────────────────────────────────────────────────────┘
                             │
                             ▼
┌────────────────────────────────────────────────────────────────┐
│ Phase 3  Local filter + enrichment  [no network cost]          │
│          For each candidate:                                   │
│          - drop if contains brand/domain                       │
│          - drop if ambiguous acronym without expansion         │
│          - classify intent (regex heuristics)                  │
│          - score 0–100 by weighted features                    │
└────────────────────────────────────────────────────────────────┘
                             │
                             ▼
┌────────────────────────────────────────────────────────────────┐
│ Phase 4  Category validation        [LLM call #2, ~$0.015]     │
│          [OPTIONAL — default on, --light to skip]              │
│          For top 15 candidates, ask:                           │
│          "Would this query primarily return results about      │
│           <user's category>, or a different industry?"         │
│          Drop rejects.                                         │
└────────────────────────────────────────────────────────────────┘
                             │
                             ▼
┌────────────────────────────────────────────────────────────────┐
│ Phase 5  Intent-diverse selection   [no network cost]          │
│          Pick top-scored candidate from each intent bucket:    │
│          - 1 commercial                                        │
│          - 1 informational                                     │
│          - 1 vertical-specific                                 │
│          Remaining validated candidates → candidatePool.       │
└────────────────────────────────────────────────────────────────┘
                             │
                             ▼
┌────────────────────────────────────────────────────────────────┐
│ Phase 6  User review                [interactive]              │
│          Show top 3 + 5 alternatives from pool.                │
│          Actions:                                              │
│          - accept → save config                                │
│          - swap [N] → replace a selected query with a pool one │
│          - edit → type custom query manually                   │
│          - regenerate → new Phase 2–5 pass (costs extra)       │
└────────────────────────────────────────────────────────────────┘
```

### Why this order

- **Brainstorm before filter** — filtering a generated list is cheap. Asking the LLM to self-filter is expensive and less reliable (models pre-filter into their own biases).
- **Local filter before validation** — cheap filters (brand, acronym, format) remove 30–50% of candidates without an API call.
- **Validation after local filter** — don't spend API budget checking candidates we've already rejected locally.
- **Intent-diverse selection after all scoring** — once all surviving candidates are scored, the pick is deterministic: highest-score-per-intent. No re-rank needed.

## 5. LLM prompts — detailed

### 5.1. Phase 2 — Brainstorm prompt

```
You are a keyword research specialist configuring an AEO tracker for a brand.

Site context:
- BRAND: {brand}
- DOMAIN: {domain}
- PAGE TITLE: {title}
- META: {metaDesc}
- H1: {h1}
- H2: {h2}
- BODY EXCERPT: {firstText}
- LANGUAGE: {lang}

Task: generate 25 UNBRANDED search queries users might type to find products
in this brand's category. Distribute them across 5 intent buckets, 5 each:

  1. COMMERCIAL — "best X 2026", "top Y tools", "Y for enterprises"
     Users ready to evaluate or buy.

  2. INFORMATIONAL — "how to do Y", "what is Y", "guide to Y"
     Users researching the problem space before evaluating products.

  3. VERTICAL-SPECIFIC — "Y for [industry]", "Y for [audience type]"
     Users from a specific industry looking for tailored solutions.

  4. PROBLEM-STATEMENT — "how to fix [pain]", "alternative to [old approach]"
     Users framing their need as a problem, not a product category.

  5. COMPARISON — "X vs Y", "alternative to Z", "cheaper than W"
     Users deciding between named options.

Hard rules:
- DO NOT include the brand name "{brand}" or its domain.
- DO NOT use acronyms with multiple industry meanings (AEO, GEO, CRO, CDP, ROI, ERP)
  WITHOUT their full expansion. E.g., write "Answer Engine Optimization" not "AEO",
  since "AEO" also means "Authorized Economic Operator" in many geographies.
- Write in the site's language ({lang}).
- 3–10 words per query.

Return STRICT JSON:
{
  "commercial": ["q1", "q2", "q3", "q4", "q5"],
  "informational": [...],
  "vertical": [...],
  "problem": [...],
  "comparison": [...]
}
```

### 5.2. Phase 4 — Category validation prompt

```
You are validating search queries for a brand's AEO tracker.

The brand's category, inferred from their site:
- BRAND: {brand}
- INFERRED CATEGORY: {inferredCategory}  (from Phase 1 site analysis)
- SAMPLE CONTEXT: {brief description from meta/h1/body}

For each query below, answer: would a typical user typing this query into
ChatGPT/Gemini/Claude expect to get results about {inferredCategory}, OR
about a different industry that shares terminology?

Queries:
{list of 15 candidates}

Return STRICT JSON. For each index 0–14:
{
  "results": [
    { "index": 0, "onCategory": true,  "confidence": "high", "note": "" },
    { "index": 1, "onCategory": false, "confidence": "high", "note": "matches 'Authorized Economic Operator' customs sector, not AI/AEO" },
    ...
  ]
}
```

### 5.3. Prompt engineering notes

- **Intent buckets explicit in the prompt** — forces the model to think in categories rather than producing variations on one theme.
- **Hard rules enumerated** — models follow explicit "DO NOT" lists more reliably than implicit preferences.
- **Acronym expansion rule** with the example — one concrete example (AEO) teaches the pattern better than an abstract rule.
- **Validation prompt asks for confidence + note** — we can surface low-confidence rejections to the user instead of silently dropping candidates.

## 6. Ranking algorithm

For each candidate query, compute features then score.

### 6.1. Features (all deterministic, no network)

| Feature | How it's computed | Contribution to score |
|---|---|---|
| `wordCount` | `q.split(/\s+/).length` | sweet spot 3–7 words: +20. Below 3: −10. Above 10: −15. |
| `hasAmbiguousAcronym` | matches against AMBIGUOUS_ACRONYMS list, AND the expansion isn't in the query | −30 (strong penalty) |
| `containsBrand` | case-insensitive substring check | −100 (hard disqualification) |
| `containsDomain` | hostname tokens present | −100 (hard disqualification) |
| `containsRecencyMarker` | `\b(2026|latest|new|recent)\b` | +5 |
| `isLongTail` | ≥ 5 words AND specificity marker present | +10 |
| `specificityMarker` | industry/audience/use-case word (heuristic list) | +10 |
| `hasComparisonStructure` | matches `\bvs\b|\balternative to\b|\bcompared to\b` | +8 |
| `languageMatch` | matches expected language from site `<html lang>` | +15 if matches, −20 if mismatched |
| `intentClassification` | rule-based: regex buckets + keyword list | no direct score, used for selection |

### 6.2. Base score + decay

```
score = 50 (base)
      + sum(feature contributions above)
      clamped to 0–100
```

Negative scores (from hard disqualifications) cause the candidate to be **dropped before ranking**, not ranked low.

### 6.3. Intent classification (Phase 3 sub-step)

Simple regex + keyword buckets:

```
COMMERCIAL:     /\b(best|top|compare|review|price|pricing|buy|purchase|choose)\b/i
INFORMATIONAL:  /\b(how to|what is|guide|tutorial|learn|explain|introduction)\b/i
VERTICAL:       /\bfor\s+(saas|enterprise|startup|small business|agency|\w+-industry)\b/i
PROBLEM:        /\b(fix|solve|stop|avoid|replace|alternative to|instead of)\b/i
COMPARISON:     /\b(vs|versus|compared|alternative|cheaper than|better than)\b/i
```

If a query matches multiple buckets, prefer the one earliest in the order above (commercial → informational → vertical → problem → comparison). This prioritises "act" over "learn" over "fit" over "solve" over "pick".

### 6.4. Selection (Phase 5)

1. Group surviving candidates by intent bucket.
2. Sort each bucket by score descending.
3. Pick top-1 from commercial, informational, vertical as the **selected triplet**.
4. If any bucket is empty: fall back to next-best in adjacent bucket (problem → vertical, comparison → commercial). Warn in UX: "No pure X-intent candidates survived; used nearest neighbour."
5. Remaining top-5 per bucket (excluding selected) → **candidate pool** persisted in config.

## 11. Testing strategy

### Unit-testable (stable, no network)

- `classifyIntent(text)` — fixtures of known queries → expected bucket.
- `scoreCandidate(text, brand, domain, lang)` — synthetic inputs, expected scores.
- `parseBrainstormResponse(raw)` — sample LLM outputs (normal, markdown-wrapped, malformed).
- `parseValidationResponse(raw)` — sample LLM validation outputs.
- `selectTopThree(candidates)` — various candidate distributions, expected triplets and fallbacks.

Target: >90% coverage on pure-function modules.

### Integration-testable (with mock LLM)

- Full `research(site, brand, domain)` pipeline with a mock LLM returning canned brainstorm + validation responses.
- Assert final selection matches expected.
- Assert candidate pool contains expected alternatives.

### Field-test checklist (before publish)

1. Run `init --auto` on `webappski.com` → verify AEO expanded, no customs results.
2. Run `init --auto` on `typelessform.com` → verify category match (voice form filling).
3. Run `init --auto` on a Polish-language site → verify Polish queries generated.
4. Run `init --auto --light` → verify fallback to single-shot behaviour.
5. Run `init --swap` → verify pool-based swap works without LLM call.

## 12. Open questions (decide before implementation)

### D1. Validation phase — default on or opt-in?

- Default on (current plan): +$0.015 per init, catches category mismatches.
- Opt-in: cheaper init by default, but accepts false positives.

**Recommendation: default on.** The $0.015 saves users from silently bad queries — classic case where opt-in means "no one opts in".

### D2. Intent-diversity strictness

- Strict (current plan): exactly 1 per intent of 3 queries — fallback to adjacent bucket if empty.
- Relaxed: pick top-3 by score regardless of intent.

**Recommendation: strict with fallback.** A tracker measuring only commercial-intent queries misses a whole dimension (informational = early-funnel users).

### D3. Candidate pool size in config

- 5 total (current plan): enough for 1–2 swaps without regenerating.
- 15 total: richer pool, bigger `.aeo-tracker.json` (~1 KB extra).

**Recommendation: 12 (4 per intent bucket × 3).** Enough depth per bucket for realistic swap scenarios.

### D4. Re-run vs refresh

- Separate commands: `aeo-tracker refresh-keywords` and `aeo-tracker refresh-selection`.
- Or: flags on `init` (`init --refresh-keywords`).

**Recommendation: flags on init.** Fewer commands, clearer mental model. Both flags bypass overwrite prompt — they're in-place refreshes.

### D5. Compatibility with 0.4.0 configs

- On first run against a v1 config (no `candidatePool`), do we:
  - (a) Silently accept and skip pool-based operations
  - (b) Warn and suggest `init --refresh-keywords`
  - (c) Force upgrade by auto-regenerating pool

**Recommendation: (b) warn + suggest.** Auto-regenerating costs the user money without consent. Silent acceptance hides that a feature is inactive.

## 13. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Brainstorm LLM returns fewer than 5 per bucket | Medium | Medium | Accept 4+ per bucket; fail hard if total surviving < 6 |
| Validation LLM rejects good candidates | Low | High | Log all rejections with notes so user can review; conservative confidence threshold |
| Regex intent classification misclassifies | Medium | Low | Human-review queries in intent diversity step; surface misclassifications visually |
| Ambiguous acronym list becomes stale | Low | Low | List lives in one file, trivial to extend via PR |
| User on CI pipeline hits $0.04 cost unexpectedly | Low | Low | `--light` flag and docs clearly state full-mode cost |
| Non-English sites produce broken queries | Medium | High | Explicit language instruction in prompt + integration test on Polish site |
| Pool-based swap UX feels clunky in terminal | Medium | Low | Keep swap flow ≤3 keystrokes, test end-to-end |
