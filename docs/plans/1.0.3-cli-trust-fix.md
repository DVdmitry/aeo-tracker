## Fix 9 ŌĆö `cmdCrawlStats` streaming parse

**File**: `bin/aeo-tracker.js:3251` (the `cmdCrawlStats` body)

**Current**:

```js
const content = await readFile(args.logFile, 'utf-8');
const entries = parseAccessLog(content);
```

A 100MB Apache log allocates 100MB+ of heap before parsing starts. On constrained CI runners this can OOM silently.

**Target**: stream-parse line-by-line using `parseLogLine` (which is **already exported** at `lib/report/log-parser.js:24` ŌĆö no refactor needed; the architect's first review caught this).

**Import style ŌĆö keep dynamic** to match the existing pattern at `:3250` (lazy-load for cold-start: `cmdCrawlStats` is rarely invoked and pulling the parser only when needed keeps every other command snappy). Switching to top-of-file static import here would be inconsistent with neighbouring commands.

```js
// inside cmdCrawlStats, replacing :3250-3253

const { parseLogLine, summariseBotCrawls } = await import('../lib/report/log-parser.js');
const { createReadStream, statSync } = await import('node:fs');
const { createInterface: createReadline } = await import('node:readline');

const size = statSync(args.logFile).size;
if (size > 100 * 1024 * 1024) {
  console.log(`${c.dim}  Streaming ${Math.round(size / 1024 / 1024)}MB log ŌĆö this may take 30-60s${c.reset}`);
}

const entries = [];
const stream = createReadStream(args.logFile, { encoding: 'utf-8' });
const rl = createReadline({ input: stream, crlfDelay: Infinity });
for await (const line of rl) {
  const entry = parseLogLine(line);
  if (entry) entries.push(entry);
}
const stats = summariseBotCrawls(entries);
```

**Note**: the existing `parseAccessLog(content)` export is left intact ŌĆö `test/log-parser.test.js` may depend on it. We just call `parseLogLine` directly from `cmdCrawlStats`.

**Why this fix and not another**: rejected alternative was "chunk-read in 10MB slices and reassemble". That doesn't solve the parser working in memory; `readline` is the standard Node idiom for line-by-line.

**Cross-platform note**: `crlfDelay: Infinity` handles CRLF line endings (Windows IIS logs occasionally have CRLF even when the user runs the CLI on macOS over an exported log).

**Regression test**: not adding a new test ŌĆö the existing `test/log-parser.test.js` covers `parseLogLine` correctness; streaming is a memory-management change, not a behaviour change. Add a one-liner comment in `cmdCrawlStats` referencing the memory expectation.

**Acceptance**: `crawl-stats` on a 500MB log file runs to completion on a 1GB-heap Node process. (Manual smoke test, not automated ŌĆö generating a 500MB synthetic log is overkill for CI.)

## Test 2 ŌĆö Static-grep invariant (glob-based, future-proof)

New file: `test/suggested-commands-resolvable.test.js`

Architect's remark: hard-coded file list re-introduces the same regression risk for the next added panel. Glob `lib/errors/*.js` and `lib/init/**/*.js` so future panels are auto-covered.

```js
// Scan every `aeo-platform init --yes` string literal in lib/errors/ and
// lib/init/ ŌĆö each must include at least one of --auto, --manual, --keywords.
// Catches the class of bug where a panel prints a suggested fix command that
// the CLI's own precondition would reject. Glob-based so new panels are
// auto-covered without remembering to update this test.

import { readFileSync, readdirSync, statSync } from 'node:fs';
import { join, relative } from 'node:path';
import { fileURLToPath } from 'node:url';
import { dirname } from 'node:path';
import test from 'node:test';
import assert from 'node:assert';

const here = dirname(fileURLToPath(import.meta.url));
const repoRoot = join(here, '..');

function walk(dir, acc = []) {
  for (const name of readdirSync(dir)) {
    const full = join(dir, name);
    const st = statSync(full);
    if (st.isDirectory()) walk(full, acc);
    else if (name.endsWith('.js')) acc.push(full);
  }
  return acc;
}

const targets = [
  ...walk(join(repoRoot, 'lib/errors')),
  ...walk(join(repoRoot, 'lib/init')),
];

const MODE_RE = /--auto|--manual|--keywords/;

test('every "aeo-platform init --yes" suggestion includes a mode flag', () => {
  const violations = [];
  for (const file of targets) {
    const src = readFileSync(file, 'utf-8');
    const lines = src.split('\n');
    for (let i = 0; i < lines.length; i++) {
      if (lines[i].includes('aeo-platform init --yes')) {
        // Look at this line + next 4 lines (suggestions may be split across template literals)
        const window = lines.slice(i, i + 5).join(' ');
        if (!MODE_RE.test(window)) {
          violations.push(`${relative(repoRoot, file)}:${i + 1} ŌĆö missing mode flag in suggestion`);
        }
      }
    }
  }
  assert.equal(violations.length, 0, `Found ${violations.length} unreachable suggestion(s):\n${violations.join('\n')}`);
});
```

Add `test:suggested-commands` to `package.json` scripts and to the top-level `test` chain.

## Risk + rollback

### Risks

1. **Fix 1 (precondition relax)** ŌĆö if `opts.keywords` is somehow truthy in a path that shouldn't reach init (e.g. a parser bug), we now skip the precondition. Mitigation: regression test asserts the error fires when `--keywords` is absent.
2. **Fix 9 (streaming crawl-stats)** ŌĆö behaviour-neutral; the streaming change uses the already-exported `parseLogLine`. Existing `test/log-parser.test.js` continues to cover parser correctness. The only new failure mode is stream-read errors on the source file (permission, EBADF mid-read) ŌĆö wrap the `for await` in a try/catch and surface as a fatal error with the source path.
3. **Fix 5 (Blocked dedupe)** ŌĆö wrong choice of "most specific classification" could mask a useful signal. Mitigation: the validator panel still prints the per-stage check counts (`LLM industry-fit check ŌĆö N flagged`), so the operator sees both stages fired even if the per-query rendering is collapsed.
4. **Fix 7 (run-manual hard-fail)** ŌĆö breaks any user workflow that intentionally runs `run-manual` with a partial directory (e.g. importing one provider's responses one query at a time over multiple invocations). Mitigation: this aligns with the `no_silent_aborts` rule (memory: `feedback_no_silent_aborts.md`); silent partial runs were the latent bug. Document in CHANGELOG under "Fixed" so users see it. **Considered**: adding `--allow-partial` as an explicit escape hatch ŌĆö deferred to 1.0.4 since no user has reported needing it; revisit only if a real workflow emerges.

### Rollback plan

**Primary**: publish `1.0.4` with the offending fix reverted. This is the realistic path; `npm publish` of a hotfix takes 5 minutes and reaches users on next `npm update`.

**Best-effort fallback**: `npm unpublish aeo-platform@1.0.3` within 72 hours, **only if** the package still meets npm's unpublish policy (no dependents, <300 weekly downloads). If `aeo-platform` has any users on `^1.0.3` carets in their `package.json`, npm denies the unpublish. Do not treat this as the primary path.

**Safety net**: `npm install aeo-platform@1.0.2` continues to resolve for anyone pinning. The deprecated package `@webappski/aeo-tracker@latest` (still at `0.3.3` redirect README) is unaffected.
